Getting the stack to run

Everything here runs without root or containers: vLLM in a Python virtual environment under a normal user account. Three findings from the bring-up, all specific to this GPU generation (compute capability 12.0):

The pip CUDA toolchain does not work for JIT kernels. The CUDA 13 wheel line ships an nvcc newer than the bundled headers of every kernel-JIT library in the serving stack (flashinfer, tilelang, deepgemm), and each rejects the pairing at compile time; the pip toolkit also omits curand headers. The system CUDA 13.0 installation is the only self-consistent JIT toolchain on the node — every serving process runs with CUDA_HOME pointed at it. Stale JIT caches from failed builds poison retries and must be cleared when the toolchain changes.

Released wheels cannot serve DeepSeek-V4-Flash on this card. Ten configurations across two vLLM wheel versions and three weight formats produced seven distinct failure modes: JIT header rejections, a scale-factor layout its GEMM library does not implement for this architecture, a missing kernel dispatch in the precompiled fallback, and an API mismatch between the engine and its attention library. A source build of vLLM (main branch) serves the model cleanly — its kernel libraries move in lockstep and carry current support for this architecture.

Long-running servers need session lingering. Processes launched over SSH die with a clean SIGTERM when the last session disconnects unless loginctl enable-linger is set for the account — both serving lanes were reaped this way overnight before the cause was found. An unprivileged-serving detail worth knowing before it costs a night.

Hardware-native FP4 works. The NVFP4 quantization of Qwen3-Coder-480B (272.9 GB) loads and serves on a 4-GPU NUMA-local group through the FP4 GEMM path, with coding capability statistically unchanged from the FP8 original (8/30 versus 9/30 on the companion node).

Serving throughput

The engine mattered more than the interconnect. On the companion 4× H200 NVL node (vLLM v0.23.0), DeepSeek-V4-Flash ran 46.7 tok/s single-stream with a 3.3-second first token. Here — PCIe instead of NVLink, and a GPU with roughly a third of the memory bandwidth — the source-built engine serves the same model at 103.5 tok/s with a 94 ms first token: 2.2× the decode speed and a 35× faster first token. The newer sparse-attention kernels, not the hardware, dominate this comparison.

Concurrent-stream capacity

Same rules as the companion report: capacity is the largest concurrency holding 20 output tokens/s per stream, fresh prompts every run; 512-token prompts with 256-token outputs (light load) and 8,192-token prompts with 512-token outputs (agent-sized). Multi-instance rows run simultaneously, one bench client per instance.

Replica scaling is exact on this node. For all three models, per-instance throughput at full box load is identical to the instance running alone — within 2% across 14 simultaneous-instance measurements. Eight single-GPU replicas, four 2-GPU tensor-parallel pairs, or two 4-GPU groups: none of them contend. This is the property the no-NVLink topology was expected to threaten and does not.

Crossing the NUMA boundary costs almost nothing for 2-GPU tensor parallelism. A DeepSeek-V4-Flash instance split across the two sockets (one GPU on each) decodes at 105.0 tok/s single-stream versus 103.5 NUMA-local, and loses 5.5% at 56 concurrent streams. Decode-time all-reduces at this scale are too small for the interconnect topology to matter.

Coding-agent evaluation

The same 30-task suite, harness, and rules as the companion report: stock-default terminal coding agent, hidden tests, zero retries, 600-second cap. One run per model.

Concurrent-stream capacity

Capacity counts concurrent streams, each held at 20 output tokens/s or better, with fresh prompts on every run. Multi-instance rows run separate model instances simultaneously (one per GPU for the 30B, one per NUMA-local GPU pair for DeepSeek, one per NUMA-local quad for the 480B) and sum the per-instance knees.

Tensor parallelism helps more here than on the NVLink node. Qwen3-Coder-30B single-stream: 132 → 156 → 158 tok/s at 1 → 2 → 4 GPUs (+18%, then a latency plateau); 8-stream aggregate: 479 → 711 → 921 (+92% at 4 GPUs). On the H200 node the same model gained only +6% single-stream from NVLink tensor parallelism. The inversion is bandwidth arithmetic: this GPU’s memory bandwidth is the bottleneck, so splitting weights across GPUs buys real speed, and the PCIe all-reduce cost is small at this model size. For capacity, replicas still beat tensor parallelism; for latency on a single stream, a 2-GPU split is worth 18%.
